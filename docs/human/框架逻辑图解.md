# 框架逻辑图解

> 一张图看清从点击"提交日记"到看到结构化事件的完整流程。技术细节在 `docs/ai/05_核心流程.md`。

## 总体结构（粗略）

```
┌──────────── 浏览器 (Vue 3 SPA) ────────────┐
│ 写日记 / 看统计 / 维护画像 / 与 AI 聊天     │
└───────────────────┬────────────────────────┘
                    │ HTTPS + JSON
                    ▼
┌──────────── FastAPI 网关 ───────────────────┐
│ 鉴权 · 参数校验 · 自动 OpenAPI              │
└───────────────────┬────────────────────────┘
      ┌─────────────┼─────────────┐
      ▼             ▼             ▼
   Service     AI Pipeline    BackgroundTasks
   业务编排   Prompt→LLM→校验→Action  异步解析
      │             │
      ▼             ▼
   Repository   LLMProvider（DeepSeek / OpenAI / Ollama）
      │
      ▼
   PostgreSQL + JSONB
```

## 日记解析一条龙

```
你按下"提交并 AI 解析"
    │
    ├─> 后端存一条 diary（status=draft）
    ├─> 立刻返回 diary_id，前端拿去展示"解析中"
    │
后台任务接手：
    1. 读你的 profile（隐私字段过滤掉）
    2. 读所有开启的模块（schema + prompt 片段）
    3. 拼出完整 prompt
    4. 查缓存（Redis），命中就跳到第 6 步
    5. 调 DeepSeek，落 ai_call_log
    6. 清洗 ```json``` 包裹 → 解析
    7. 按每个模块的 schema 校验
         单条坏 → 只重做那一条（最多 3 次）
         整体坏 → 标 failed 结束
    8. 把 AI 产出的 actions 进事务执行
         每条 action 落 action_log
         任何一条失败 → 整篇回滚、标 failed
    9. 全部成功 → diary.status = parsed
         ⚠️ 当前前端要刷新才看到（待修）
```

## 模块怎么"插"进来的

每个模块 = 一个小目录 + 三件套：

```
server/app/modules/expense/
  ├── schema.py     ← 告诉系统"消费事件长什么样"（Pydantic）
  ├── prompt.py     ← 告诉 AI "日记里看到花钱就按这个提取"
  └── module.py     ← 继承 _base.Module，@register 一下
```

后端启动时自动扫描注册，不需要改任何主流程代码。

## 三层纠错防线

```
第 1 层：LLM 吐错 JSON
   → schema 校验失败 → 把错误塞回 prompt 让它重写（最多 3 次）

第 2 层：入库执行出错
   → 整个事务回滚 → diary 标 failed → 你可以点"重试"

第 3 层：你改过的事件
   → event.locked = true → 之后无论怎么重解析，这条都不动
```

## 权限

- 注册/登录发 JWT，前端存 localStorage
- 所有 API 都强制按 `user_id` 隔离（即使你写错了 service，也不会串用户数据）
