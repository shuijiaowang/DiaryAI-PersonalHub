## 近期（1~2 周内值得做）

### 1. 修"解析完不自动刷新"的 bug（优先级最高）

- 现象：提交日记后前端停留在"解析中"，必须手动刷新 + 再点一次"解析"才看到结果
- 方案：`DiaryDetailView` 里加 3~5 秒轮询 `GET /api/v1/diaries/{id}`，收到 `parsed / failed` 停表
- 远期：换成 SSE 或 WebSocket 推送
- 这个不修，后面任何"解析体验"相关的测试都会假阳性

### 2. 事件编辑 UI

- 详情页每条 event 旁加"编辑 / 删除 / 锁定"按钮
- 编辑提交后落到 `event.data`，同时把 `locked = true`
- 后端 API 已有，纯前端工作量

### 3. 统计页补完

- ✅ 消费统计独立页：日/周/月/年切换 + 类型饼图 + 日历热力图 + 明细表（2026-04-20）
- ⬜ 饮食模块补一个"次数 + 花费"双轴图（按 `ExpenseStatsView` 的模板做 `MealStatsView`）
- 用 ECharts 的 `LineChart` + `BarChart`（已接入 `vue-echarts`）

### 4. 日记搜索

- 先做 `raw_text ILIKE %keyword%`（5 分钟能上）
- 后续升级成 PostgreSQL 的 `tsvector` 全文检索

### 5. 备忘录闭环 e2e 测试

- 写一条"明天买抽纸" → 第二天日记写"今天买了抽纸" → 断言第一条 `done = true`
- 放 `server/tests/e2e_with_real_llm/`（或 mock LLM 版本放常规 tests）

## 中期（1~2 个月）

### 6. 把 BackgroundTasks 换成 Celery（当出现以下信号时才做）

触发条件：
- 单次解析超过 30 秒导致请求阻塞
- 想加"定时任务"（日报/周报生成）
- 多用户后并发解析把进程拖死

不满足就别折腾，按 ADR-005 的决定继续 BackgroundTasks。

### 7. 前端状态管理整理

- 目前 Pinia 的 store 相对简单，后面加"实时通知"/"多标签联动"会不够用
- 考虑加一个 `realtime` store，轮询或 WS 统一挂这

### 8. 第 5 个内置模块

候选：`reading`（阅读）/ `game`（游戏）/ `exercise`（运动）/ `mood`（情绪）
挑一个自己最常写的开搞，走一遍"加模块 = 加目录 + @register"的流程把经验沉淀到 `docs/ai/04_模块约定.md`。

### 9. 数据导出

- 一键导出某用户的全部 diary + event 为 JSON / Markdown
- 方便迁移 / 备份 / 给 AI 做离线分析
