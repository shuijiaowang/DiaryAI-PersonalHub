# DiaryAI-PersonalHub · 项目文档（需求与想法）

> 本文档只描述「想法 / 需求 / 边界」，不涉及任何技术选型与具体实现。技术与架构请看 `设计文档_ai.md`。

---

## 0. 一句话定位

> **以"每日日记"为唯一输入入口，由 AI 把自然语言切片成结构化数据，沉淀成一份持续生长的"个人数字档案"，并在档案之上提供统计、提醒、陪伴和自动化能力。**

---

## 1. 立项动机（Why）

随着 LLM 能力爆发，未来的应用形态一定会从"用户去操作软件"转变为"AI 代理用户接管数据并主动服务"。我希望先做一个**最小可行的雏形**：

- 短期：每天写一篇日记，AI 自动把消费、饮食、运动、阅读、游戏、人际关系等抽出来，前端给我看日 / 周 / 月 / 年的统计；
- 中期：除了"被动写日记"，还能从外部数据源（外卖、消费记录、Steam、微信读书等）**被动监听**输入；
- 长期：AI 拿到这份越来越完整的"用户画像"后，可以做：
  - 主动建议（"你这个月外卖支出比上月多 30%，要不要做饭？"）
  - 情感陪伴（基于性格和最近事件聊天）
  - 自动化代办（自动添加备忘、自动生成礼物清单、自动安排日程）
  - 动态 UI（不同人看到的首页不一样，按当前最关心的事件展示）

所以这个项目本质上是 **"个人数据中枢（Personal Hub）+ AI Agent 入口"**，而不是一个普通的日记 App。

---

## 2. 用户与场景

### 2.1 目标用户
- 第一阶段：**自己一个人用**（单用户/本地优先）。
- 第二阶段：可注册多用户，每个人拥有独立的画像和数据，互相隔离。

### 2.2 核心使用场景

| # | 场景 | 用户做什么 | 系统做什么 |
|---|---|---|---|
| 1 | 写日记 | 在前端文本框里写一段当天的自然语言流水账 | AI 解析 → 抽事件 → 入库 → 更新统计 |
| 2 | 看统计 | 打开"消费 / 饮食 / 阅读 / 游戏"等模块页 | 展示日 / 月 / 年图表与明细 |
| 3 | 维护画像 | 在"我的资料"里补全或修改全局信息（住址、工作、兴趣等） | 这些信息会被自动喂给 AI，作为后续解析的上下文 |
| 4 | 备忘 / 提醒 | 在日记里随手写一句"明天要买抽纸" | AI 自动写入备忘录模块；完成后下次日记中提到则自动标记完成 |
| 5 | 回看与编辑 | 浏览历史日记 / 修改 AI 解析错误的事件 | 支持人工修正，修正结果纳入训练上下文 |
| 6 | 跟 AI 对话 | 直接问"我上周吃了多少次面？" | AI 基于结构化数据回答 |

## 3. 核心概念（领域模型，文字版）

### 3.1 日记（Diary）
- 一天**最多一篇**（同一天再写视为追加 / 重新解析）。
- 包含：原始文本、AI 规范化后的文本、解析得到的事件列表、状态（未解析 / 解析中 / 解析失败 / 已确认）。

### 3.2 事件（Event）
- AI 从日记里切出来的最小数据单元。
- 每个事件挂在某种"事件类型（模块）"下，例如：天气、饮食、消费、运动、阅读、游戏、社交、情绪、健康、工作……
- 字段统一为：
  - `raw_text`：日记里对应的原文片段（不可改写）
  - `ai_processed_text`：AI 规范化后的描述
  - `data`：该事件类型专属的结构化字段（扁平 + 一个自由发挥的 `extra` 子字段）

### 3.3 全局信息（GlobalInfo / Profile）
- 描述"我是谁"的长期数据，例如：基本信息、地址、工作、家庭、人际关系、兴趣爱好、成就、玩过的游戏、看过的影视、收藏夹、备忘录等。
- 这部分是**模块化**的，每个模块本质都是 "schema + 用户配置 + 实际数据" 三件套。
- 它**既是 AI 的输入**（让 AI 更懂我），**也是 AI 的输出**（AI 解析日记后会反过来更新这些模块）。

### 3.4 模块（Module）
- 系统级概念。无论是"事件类型"还是"全局信息分类"，都用模块来统一表达。
- 模块分两种：
  - **内置模块**：天气、饮食、消费、备忘录…… 系统自带。
  - **插件模块**：未来可以让用户/AI 自己定义新模块（例如"我想加一个'追番'模块"）。
- 每个模块对外暴露三件事：
  1. **schema**：数据长什么样（给 AI 当模板）
  2. **prompt 片段**：怎么从日记里提取它
  3. **action**：AI 解析后允许执行哪些动作（insert / update / delete / append…）

### 3.5 动作（Action）
- AI 解析后不直接写库，而是产出一组"动作描述"，例如：
  ```
  { action: "insert", module: "备忘录", value: { content: "买生日礼物", due: "2026-3-8" } }
  { action: "update", module: "备忘录", target_id: 3, value: { done: true } }
  ```
- 系统统一调度执行 → 入库 → 失败可回滚或交给 AI 重做（见第 5 章纠错系统）。

## 4. 功能清单（What）

### 4.1 必做（MVP，第一阶段）
1. 用户注册 / 登录 / 鉴权
2. 写日记（按日期，一天一篇，可追加重解析）
3. AI 解析：日记 → 事件列表（至少支持：天气 / 饮食 / 消费 三类）
4. 事件人工修正（编辑、删除、补充）
5. 全局信息维护（基本信息、地址、工作、人际关系、兴趣 …… 模块化）
6. 备忘录模块：日记里写"要做的事" → 自动入备忘录；后续提到完成 → 自动勾掉
7. 简单统计页：消费日/月/年柱状图、饮食次数与花费、最近事件流
8. 历史日记浏览与搜索

### 4.2 应做（第二阶段）
- 模块管理后台（启用/禁用模块、编辑 schema 与 prompt）
- 多模块解析（运动、阅读、游戏、社交、情绪等）
- 与 AI 对话查询数据（"我这周咖啡花了多少钱？"）
- 日报 / 周报 / 月报自动生成
- 数据导出（JSON / Markdown）

### 4.3 想做（远期）
- 插件市场：第三方贡献模块
- 外部数据被动接入：外卖、银行账单、Steam、微信读书、网易云、健康数据……
- 主动建议与自动化（提醒、消费预警、生日提醒、礼物推荐）
- 情感陪伴模式（基于画像的角色化对话）
- 多端：Web + 移动端 + 桌面端
- 端侧 / 私有部署，数据完全本地化

## 5. 关键设计原则（不是技术，是产品原则）

1. **AI 是后端的一部分，不是外挂**
   AI 调用必须像数据库一样可重试、可审计、可回滚。任何 AI 输出在落库前必须经过 schema 校验。

2. **模块化优先**
   宁可前期多写一份模板，也不要把字段硬编码进业务代码里。新加一个模块 = 新加一份 schema + prompt 片段，不需要改前端框架。

3. **原文是真理**
   `raw_text` 永远保留原始日记片段，AI 解析结果可以重做、可以纠错，但原文不可丢失。

4. **可纠错 / 可回滚**
   - 单条事件解析错误：可让 AI 单独重做这一条（不重跑整篇）
   - 整体执行失败：整篇回滚，状态标记为"解析失败"，等用户重试
   - 用户人工修改过的事件：之后 AI 重解析时不能覆盖（要打"已锁定"标记）

5. **隐私与数据自主**
   - 必须支持"私密模块"（账号密码、健康隐私等不发给云端 AI）
   - 全局信息字段级别可标"是否参与 AI 上下文"
   - 长期目标支持本地大模型，关键数据不出端

6. **小步快跑，模板先行**
   先用配置文件 / JSON 模板把流程跑通（项目当前就是这个状态），再逐步把它"产品化"成数据库 + 后台 + 前端。

---

## 6. 边界与不做的事

- ❌ 不做社交：这不是朋友圈、不是 Day One 的分享功能。日记默认私有。
- ❌ 不做富文本编辑器：MVP 就是纯文本输入框。
- ❌ 不做复杂权限体系：每个用户只能看自己的数据。
- ❌ MVP 不接入外部数据源：先把"日记 → 结构化"这条主线打磨好。

## 7. 当前已有产出（继承自老代码）

虽然项目要重构，但以下资产是已经验证可用的，新版本必须复用思路：

1. **Prompt 工程链路**：`input.json` 模板 + `templates.jsonc`（事件 schema）+ `output.jsonc`（输出格式）+ `global_info.json`（用户画像）→ 拼成完整 prompt → DeepSeek → 清理 \`\`\`json 包裹 → 解析为 JSON。
2. **事件模板示例**：天气 / 饮食 / 消费 三类的字段设计可直接迁移。
3. **action 设计草稿**：备忘录的 `insert / update` action 模式（见 `global_model_memo.jsonc`）可以推广到所有模块。
4. **数据库表的雏形**：User / Diary / Event / GlobalModule / GlobalContent 五张表的字段思路（见 `model/数据库设计.md`）可以作为新 schema 的起点。
5. **样例数据**：`diaries/2026-3-5.txt` + `result/output.json` 是一对完整的 输入 → 输出 真实样本，可以直接拿来当端到端测试用例。

## 8. 验收标准（MVP 完成的判定）

只要满足以下场景就算 MVP 跑通：

> 我打开网页 → 注册账号 → 在"我的资料"里填上基本信息和女朋友昵称 → 在日记页粘贴 `2026-3-5.txt` 里那段文字 → 点"AI 解析" → 几秒后看到三类事件被正确切出来 → 跳到"消费"页能看到当天 700+ 元的支出明细 → 跳到"备忘录"看到"洗澡 / 打扫 / 买抽纸"被自动添加为待办 → 第二天再写日记说"今天买了抽纸"，备忘录里那条自动被勾掉。

完成这个闭环 = MVP 成功。
