# 08 · 决策记录（ADR 轻量版）

> 重要技术决策的留痕。目的：**避免 AI 反复推翻同一件事**。
> 每条格式：日期 / 决策 / 背景 / 备选 / 生效范围。

---

## ADR-001 · 后端用 Python + FastAPI（2026-04 立项）

**决策**：后端 Python 3.11+，Web 框架 FastAPI。

**背景**：项目瓶颈是 LLM 而非 QPS；老代码已有 Python + DeepSeek 链路；AI 写 Python 训练数据多。

**备选**：Go（性能好但 LLM 生态弱）、Node.js（可行但 data/AI 生态弱）。

**范围**：不再讨论换后端语言。如果未来要加 BFF/SSR 可以叠一层 Node，但主后端不动。

---

## ADR-002 · 数据库用 PostgreSQL（2026-04）

**决策**：PostgreSQL 16+，`event.data` / `module.schema` / `profile_section.content` 用 JSONB。

**背景**：80% 结构化 + 20% 半结构化；JSONB 可索引可查询；远期向量检索直接 pgvector。

**备选**：MySQL（JSON 弱）、MongoDB（关系表达弱）。

**范围**：不再讨论换数据库。

---

## ADR-003 · LLM 调用必须过 `LLMProvider` 抽象（2026-04）

**决策**：所有模型调用走 `app/ai/providers/base.py:LLMProvider`；新增模型 = 新加子类 + 注册。

**背景**：要能随时切换 DeepSeek / OpenAI / Claude / Ollama；要统一做缓存 + 审计。

**范围**：业务代码禁止直接 `import openai` 或裸 HTTP 调模型。已在 rules 第 1 条硬化。

---

## ADR-004 · 模块化 = 三件套 + `@register`（2026-04）

**决策**：每个业务/画像模块 = `code + name + DataSchema + prompt_fragment + actions`，通过 `app/modules/registry.py:@register` 自动注册。主流程禁止 `if module_code == "xxx"`。

**背景**：要能"加模块不改主流程"；要能热加载用户自定义模块（远期）。

**范围**：新加任何类型（运动、阅读、游戏、社交、情绪……）都走这套。

---

## ADR-005 · 异步任务先用 FastAPI BackgroundTasks（2026-04）

**决策**：MVP 阶段日记解析走 FastAPI `BackgroundTasks`；前端轮询拿结果。Celery / ARQ 推迟到"用户量/并发压力出现"再上。

**背景**：单用户自用阶段，上 Celery 运维成本 > 收益；Redis 也设为可选。

**范围**：解析超时 / 多任务并发问题**不算**重构触发条件；只有明确出现性能瓶颈才切。切换时走 ADR 新增。

---

## ADR-007 · 日记原文不可改写，用户补录走 locked event（2026-04-20）

**决策**：
- `diary.raw_text` 一旦写入**永不被系统修改**（不追加、不覆盖、不改写、不弹窗问用户）。
- 用户在网页上手动添加 / 修改 / 补充的条目 → 只写 `event` 表，并**自动 `locked=True`**。
- 手动事件挂到"当天的 diary"上。如果那天没有日记，Service 自动建一个空壳 Diary（`raw_text=""`, `status=parsed`）作为外键归属。
- 重解析（已有逻辑）只 `delete_unlocked_by_diary`，所以 locked 事件天然被保留。

**背景**：
- 日记是真相之源，任何系统写回都会让「原文」变成一个模糊概念。
- 追加到末尾会污染原文 → prompt 体积涨、重解析得区分哪段是"原文"哪段是"系统追加"，工程复杂度 >> 收益。
- "弹窗让用户选改原文 / 追加 / 仅记事件"会多出三条路径，每条路径都有边界条件（锁定覆盖、历史事件归属、重解析策略），不划算。
- `event.locked` 这一层已经是完备的"用户修正"接口，用它承载手动补录就够了。

**备选**：
- A. 在日记末尾按格式追加"[系统补录] 2026-04-20 10:00 晚饭 45 元"，再次解析时识别这些段落跳过——**否决**（理由见背景）。
- B. 引入独立 `manual_entry` 表——**否决**（和 `event` 字段几乎一致，造成双写 + 统计要 UNION）。
- C. 当前方案（复用 `event.locked`）——**采纳**。

**范围**：
- 任何模块（消费 / 饮食 / 备忘录 / 后续新增）的"前端手动补录"都走 `POST /api/v1/events`，Service 负责 lock + 归属 diary。
- 日记解析主流程 / 前端"重新解析"按钮的行为**不变**：只重建非 locked 事件。
- 空壳 Diary 在列表页可以识别并隐藏（目前先不做，等用户真的困扰再加过滤）。
- 此决策关闭以下讨论："要不要把手动记录写回日记原文"、"要不要做弹窗询问策略"、"手动记录该放哪张表"。

## ADR-006 · docs 目录拆 ai/ 与 human/（2026-04-20）

**决策**：`docs/` 下分 `ai/`（结构化、简洁、token 友好）和 `human/`（通俗、开发者自用）两个子目录。AI 工作流和文档同步规则写进 `.cursor/rules/diaryai-rules.mdc` 第 15~17 条。

**背景**：原先 5 份文档定位混乱（`_ai` 后缀但也给人看、密钥明文、进度散落在"运行状态"里）。需要让 AI 读得快、人看得懂、且每阶段结束有强制同步点。

**范围**：所有架构/设计/进度类文档只能放 `docs/ai/` 或 `docs/human/`；不允许散落到 `server/README_xxx.md` 等位置。
