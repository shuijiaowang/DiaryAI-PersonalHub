# 06 · AI 工作流 SOP

> 这是你（AI）在本仓库接到任务后的标准作业流程。**省 token 的关键不是"少读"，而是"只读对的"**。

## 开工前必做三件事

1. **读 `07_进度与状态.md`** — 知道当前到哪一步、哪些是 WIP、哪些是已知坑。避免重复实现或踩已记录的坑。
2. **根据任务类型去 `README.md` 的"按任务选读"表**挑该读的文件，只读这些。不要无差别读全 `docs/ai/`。
3. **读 `.cursor/rules/diaryai-rules.mdc`**（始终加载，不用额外读）——是硬约束边界。

## 任务类型 → 最小读 → 改完同步

| 任务类型 | 最小上下文 | 完工后必须更新 |
|---|---|---|
| 加/改模块 | `04` + 对应模块目录 | `04` 模块表 + `07` + `docs/human/开发进度.md`（如果是里程碑） |
| 加/改数据库表 | `03` + `02` 分层 | `03` + `docs/human/数据库一览.md` + Alembic 迁移 |
| 加/改 API 路由 | `02` + `03`（相关表） | 通常不必更新 docs，除非新增了"用户可见功能" → `docs/human/开发进度.md` |
| 改 AI pipeline / prompt | `05` + `04` | 如果改变了流程结构 → `05`；若是范式级变更 → `08` |
| 修 bug | `05`（如果涉及主流程）+ 相关代码 | `07` 里把 bug 从"已知"勾掉 |
| 重大方案变更（换库、废弃模式） | `08` + 受影响的 `02/03/05` | 先写 `08` 决策 → 再动代码 → 更新受影响文档 |

## 强制产出

完成任意一个上面列的任务后，**在回复用户前**必须：

1. 写完代码，确认能通过相关测试（`pytest server/tests/`）
2. 更新上表第三列指定的文档
3. 如果动了 `docs/ai/*`，自检该文件是否 ≤ 200 行、是否还有过时内容可删

**不更新文档 = 任务没做完**。

## 写代码时的默认姿势

- 新增/修改 SQLAlchemy 模型 → 必须生成 Alembic 迁移
- 新加 Pydantic schema → 放 `app/schemas/`，不要和 ORM model 混
- 新加 service 方法 → 必须在 service 层开事务（`with db.begin()` 或依赖 FastAPI dep）
- LLM 相关改动 → 确认走的是 `LLMProvider` 抽象，没绕过 audit / cache
- 加新配置项 → 加到 `Settings` + `server/.env.example`（.env 不要动）

## 禁区（踩了直接要求重做）

- 路由里直接操作 ORM / 写 SQL
- 业务代码 `import openai` 或 `requests.post("https://api.deepseek.com/...")`
- 源码出现明文密钥
- 为某个模块在主流程加 `if module_code == "xxx"`
- `Base.metadata.create_all`、手写 DDL、跳过 Alembic
- 删掉 `raw_text` 原文 / 忘记处理 `locked=true` 的 event

## token 节省技巧

- 相同信息只在一处维护（本目录 README 的表就是导航，别复制粘贴到每个文件开头）
- 长示例 JSON 放到 `server/tests/fixtures/`，文档里只写"见 fixtures/xxx.json"
- 不要在 `docs/ai/` 写"为什么不选 X"的论证——这是 `docs/human/` 或 `08_决策记录.md` 的事
