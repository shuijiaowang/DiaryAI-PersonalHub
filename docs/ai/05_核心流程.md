# 05 · 核心流程

## 1. 日记解析流程（主闭环）

```
用户 POST /api/v1/diaries
  ↓
DiaryService.create_or_upsert()
  - (user_id, date) 存在则更新 raw_text，status = draft
  - 触发 parse（本期同步 / BackgroundTasks；未来 Celery）
  ↓
DiaryParserPipeline.run(diary_id):
  1. 取 user 的 profile_section（过滤 privacy != private）
  2. 取所有 enabled 的 module 的 schema + prompt_fragment
  3. 如果 `raw_text` 末尾含"系统修改记录"区块，先剥离该区块，再用正文拼装 prompt
  4. Jinja2 拼装主 prompt
  5. prompt_hash 查 Redis 缓存（LLM_CACHE_ENABLED）
     命中 → 直接拿 response
     未命中 → LLMProvider.chat() → 写 ai_call_log → 写缓存
  6. 清洗 ```json``` 包裹 → json.loads
  7. 按 module DataSchema 逐条校验
     - 单条失败 → 仅该条回灌 error 给 LLM 重做，最多 3 次
     - 整体失败 → diary.status = failed，parse_error 记录
  8. ActionExecutor.run(actions)
     - 单事务；每个 action 落 action_log
     - 先清 unlocked 事件，保留 locked 事件
     - 新提取事件写入前与已 locked 事件做等价去重（同模块 + 同 data）
     - 失败 → 全篇回滚 → status = failed
  9. 成功 → diary.status = parsed
  ↓
前端轮询 GET /api/v1/diaries/{id} 拿最新状态
（未来替换为 WebSocket 推送）
```

## 2. Action 执行

支持的 action（由模块的 `actions` 字段声明）：

| action | 行为 | 典型用例 |
|---|---|---|
| `insert` | 新建 event 行 | 所有模块默认 |
| `update` | 按 `target_id` 更新已有 event | 备忘录标记完成 |
| `delete` | 软删除 event | 备忘录撤销 |
| `append` | 在现有 event.data.extra 里追加 | 远期 |

执行顺序：
1. 按模块声明校验 action 合法性（不在 `actions` 白名单直接拒）
2. 被作用的 event 如果 `locked=true` → 跳过并记 warning 到 action_log
3. 事务内依次执行；任意一条异常整体 rollback
4. 每条 action 无论成功失败都写 `action_log`

## 3. 纠错与回滚（三层防线）

1. **Schema 校验层**：LLM 输出不符 schema → 重试最多 3 次，每次把"上次错误 + 期望格式"塞回 prompt
2. **Action 执行层**：事务回滚 + diary 标 failed，前端可点"重试"
3. **用户修正层**：event 上打 `locked=true`，后续 AI 重解析跳过该条

## 4. 重解析

同日期再次提交日记 = 视为重解析：
- 保留 `locked=true` 的 event 不动
- 其余 event 删除后重走 pipeline
- `raw_text` 里的"系统修改记录"区块不会参与 LLM 解析（避免重复生效）
- 旧 action_log 保留（审计）

## 4.1 前端图形化修改后的原文策略（MVP）

- 手动新增 / 更新 / 删除事件时，Service 会在对应日记 `raw_text` 末尾追加自然语言"系统修改记录"。
- 该记录是用户可见变更日志，不直接改写正文原句。

## 5. 鉴权

- 注册/登录 → 颁发 JWT（access + refresh）
- `app/api/deps.py:get_current_user` 从 Authorization header 解码
- 所有查询强制按 `user_id` 隔离（Repository 层就传 user_id）

## 6. 统计页（stats）

- 当前实现：`stats_service.py` 按 `module_code + user_id + 时间范围` 聚合 `data` 里的金额/次数
- 聚合 SQL 直接用 PG 的 JSONB 运算符（`data->>'price'::numeric`）
- 未来数据量大后转物化视图
